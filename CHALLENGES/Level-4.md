> *"The best way to learn open source is to do open source — not just in DevQuest, but out there in the wild."*

---

## 🎯 Objective

Go beyond DevQuest and contribute to **another open source project**. Explore the broader open source ecosystem, find a project you care about, and make a meaningful contribution.

**Prerequisite**: Complete [Level 3 — First Merged PR](Level-3.md).

---

## 📋 Requirements

To complete Level 4, you must do **at least TWO** of the following:

- [ ] **Contribute to an external open source project** (PR merged or accepted)
- [ ] **Report a well-documented bug** on any public GitHub repository (issue closed/acknowledged)
- [ ] **Star and fork 5+ repositories** relevant to your interests and document why
- [ ] **Participate in a GitHub Discussion** on a major project (5+ helpful replies)
- [ ] **Improve documentation** on any public project (merged PR)
- [ ] **Translate** documentation or content into another language

## 📖 Finding the Right Project

### Where to Look

| Platform | URL | Best For |
|----------|-----|---------|
| GitHub Explore | [github.com/explore](https://github.com/explore) | General discovery |
| Good First Issues | [goodfirstissue.dev](https://goodfirstissue.dev) | Beginner-friendly issues |
| Up For Grabs | [up-for-grabs.net](https://up-for-grabs.net) | Curated issues |
| First Timers Only | [firsttimersonly.com](https://firsttimersonly.com) | First-time contributors |
| CodeTriage | [codetriage.com](https://codetriage.com) | Triaging issues |

### How to Choose a Project

```
Ask yourself:
 ✅ Do I use this project?
 ✅ Is there active maintainer engagement?
 ✅ Are there issues labeled "good first issue" or "help wanted"?
 ✅ Is the CONTRIBUTING.md clear?
 ✅ Is the community welcoming?
```

### Reading a CONTRIBUTING.md

Before contributing to any project:
1. Read their `CONTRIBUTING.md` fully
2. Check their PR requirements
3. Look at recently merged PRs for style guidance
4. Follow their issue/PR templates exactly

## 📖 Step-by-Step Instructions

### Task 1: External Contribution

1. Find a project with a `good first issue` label
2. Comment on the issue: *"I'd like to work on this — any guidance on approach?"*
3. Wait for maintainer acknowledgment before starting
4. Fork, branch, implement, PR — same workflow as DevQuest
5. Screenshot your merged PR (you'll need this for verification)

### Task 2: Bug Report

Write a quality bug report:

```markdown
**Bug Description**: [Clear, concise description]

**Steps to Reproduce**:
1. Go to...
2. Click on...
3. See error

**Expected Behavior**: [What should happen]

**Actual Behavior**: [What actually happens]

**Environment**:
- OS: macOS 14.0
- Browser: Chrome 120
- Version: 2.1.4

**Screenshots**: [If applicable]
```

### Task 3: GitHub Discussions

Find a project's Discussions tab and:
- Answer questions you know the answer to
- Ask thoughtful questions
- Share helpful resources
- Be positive and constructive

## 💡 Tips

> **Tip 1**: Start with projects you already use. Familiarity makes it much easier to understand the codebase.

> **Tip 2**: Documentation contributions are underrated and often faster to merge than code changes. Don't underestimate them!

> **Tip 3**: Comment on issues before submitting PRs — maintainers appreciate knowing someone is working on something before a surprise PR arrives.

> **Tip 4**: Small, focused PRs are better than large ones. Fix one bug, improve one page.

> **Tip 5**: It's okay if your first external PR isn't merged. The experience of going through the process is valuable regardless.
